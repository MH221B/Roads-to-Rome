05 Step 1: The Trigger

.github/workflows/ci-cd.yaml 
name: CI / CD
on:
  push:
    branches: [main, dev]
  pull_request:
    branches: [main, dev]
AUTOMATED START

The pipeline activates instantly when code is pushed to main (production) or dev (staging). No manual buttons required.

06 Step 2: Quality Gates

ci-cd.yaml (Server Test Job) 
  server-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci --include=dev
      - run: npm test # Blocks deployment on failure

Stop the Line

We install dev dependencies (including test runners) and execute the test suite. If npm test fails, the pipeline halts immediately. Broken code is prevented from reaching the deployment stage.

07 Step 3: Container Strategy

Dockerfile 
# 1. BUILDER (Heavy)
FROM node:20-alpine AS builder
COPY package*.json ./
RUN npm ci --silent
COPY src ./src
RUN npm run build

# 2. RUNNER (Light)
FROM node:20-alpine AS runner
ENV NODE_ENV=production
COPY --from=builder /app/dist ./dist
CMD ["node", "dist/index.js"]

Multi-Stage Build

We use a Multi-Stage Build. The first stage installs all dependencies and compiles TypeScript.

The second stage copies only the compiled artifacts (`dist`) and production modules, resulting in a tiny, secure image.

08 Step 4: Deployment

Render Hook (Backend) 
server-deploy:
  needs: [server-lint, server-test]
  run: curl -X POST ${{ secrets.RENDER_HOOK }}

Only runs if Lint AND Test jobs pass. Triggers Render to pull the Docker image and rebuild.

Vercel Config (Frontend) 
{ 
  "rewrites": [
    { "source": "(.*)", "destination": "/index.html" }
  ] 
}

Ensures SPA routing works by redirecting all traffic to index.

09 Local Development

docker-compose.override.yml 
services:
  server:
    command: npm run dev
    volumes:
      - ./:/app:delegated  # Maps source code into container

By using an override file locally, we enable Hot Reloading inside the container environment. The volume maps your local code directly into the running container.
